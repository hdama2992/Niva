Niva closed beta to Play Store

Production Integration, Testing, And Android Release Guide

Niva already has the closed-beta product code: onboarding, trusted joins, selfie review, events, circles, approved-member chat, host tools, feedback, continuity, realtime updates, and admin operations. This guide covers the remaining work that turns that code into a secure, testable Android app that can be released through Google Play.

Repository baseline: main at 4a74ab6, with the release-completion work documented here applied locally. Reviewed July 17, 2026. This is an implementation and release checklist, not legal advice.

1. Honest Current Snapshot

The core product is not a front-end mock. User, profile, trust, verification-review, event, circle, membership, chat, feedback, notification, and audit records are persisted through the NestJS API in PostgreSQL. The mobile app and dashboard call that API.

Member experience

Profile, explicit women-centered self-declaration, browse, recommendations, verified join restrictions, join-time selfie submission, host approval, cohort chat, icebreakers, feedback, settings, blocks, and continuity preferences are implemented.

Host and admin experience

Host approval, create/edit/cancel activities, foreground-assisted venue coordinates, attendance, member decisions, feedback insights, named-admin verification review, deletion-request operations, audited corrections/cancellations, and aggregate analytics are implemented.

Security foundations

Firebase ID tokens are verified by the backend. Storage paths are owner-scoped. Verification selfies are private to clients and viewed by administrators through short-lived signed URLs.

Realtime behavior

Approved event/circle members receive scoped Socket.IO chat, membership/activity updates, and in-app notification events. There are no random direct messages.

Do not treat the current Expo browser preview as a production test. It is useful for UI and local API work, but native phone authentication, notification permission/token registration, foreground location, image upload, and Android PNV still need configured providers and a physical Android development or release build.

2. Core Features In The App: What They Do And How To Test Them

This is the product path as it exists today. The application is intentionally a safe community workflow, not an open social network.

Access boundaries are deliberate: browsing is open after sign-in; joining requires profile completion, verification and guidelines acceptance; chat and approved-member icebreakers require membership approval. No feature grants a random direct-message path.

Phone sign-in and restored session

Member experience: enter phone number, confirm Firebase SMS OTP, then return to the app already signed in on the next launch.

How it works: Firebase Auth proves the phone number and gives the app an ID token and refresh token. SecureStore holds the refresh token. NestJS verifies every Firebase ID token and upserts the PostgreSQL user.

Test: sign in with a test and real number, force-close the app, reopen it, and confirm the session restores. Try an invalid/expired code and confirm no API session is created.

Profile and women-centered self-declaration

Member experience: choose username, name, city, age range, languages, interests, optional occupation/bio, profile photo, and explicitly confirm eligibility for the women-centered community.

How it works: the mobile form writes to the Users API; server validation requires a valid username, display name, city, and at least three interests. The declaration is a persisted consent record, not automated gender detection.

Test: try duplicate/invalid usernames and incomplete profiles. Complete a valid profile, restart the app, and confirm all data persists.

Join-time selfie verification and trust

Member experience: browsing works first. On the first join attempt, an unverified member is asked to upload/take a selfie and sees a pending state until an admin decides.

How it works: the image uploads to a private Firebase Storage path owned by the Firebase UID. The API records a pending verification review. An admin approves, holds, or rejects it; the API updates trust events, score, tier, and join permission.

Test: join as an unverified member, submit an image, try the Storage URL from another account, then approve/reject in admin. Confirm only approval unlocks joining and the signed viewer URL expires.

Home, explore, search, and recommendations

Member experience: a purposeful dashboard shows nearby activities, her circles, recommended events, workshops, notifications, and plans instead of an infinite social feed.

How it works: the API queries persisted events/circles and calculates recommendations using city, interests, memberships, blocks, continuity choices, and settings. Searches and filters call the backend rather than filtering only a static device list.

Test: use two profiles with different cities/interests, change an interest and a block, refresh Explore, and confirm recommendations and exclusions update from the server.

Events, circles, capacity, and join workflow

Member experience: see details, date/time, capacity, host, mutual-interest icebreakers after approval, and a clear join/leave state. My Plans groups upcoming, past, and cancelled items.

How it works: NestJS persists events/circles and memberships. It checks verification, guidelines acceptance, capacity, blocks, and lifecycle state before accepting a join request. A host then approves or declines the member.

Test: fill an activity, attempt another join from a second device, join/leave repeatedly, cancel an activity, and verify the server prevents invalid states even if a mobile request is replayed.

Community guidelines gate

Member experience: before the first authorized join, the member accepts the Niva community guidelines once.

How it works: the API persists the accepted guideline version and timestamp. The join endpoint rejects an otherwise eligible user who has not accepted it.

Test: call a join as a verified account without acceptance and confirm it fails. Accept guidelines, retry, and inspect the stored acceptance record.

Approved cohort chat and realtime updates

Member experience: after host approval, members can chat inside that event/circle. They see new messages, member decisions, edits/cancellations, and in-app notification updates without manually refreshing.

How it works: chat history is stored in PostgreSQL. Socket.IO verifies a Firebase token and authorizes a member into only her activity room. The backend persists first, then broadcasts the update.

Test: use two approved devices and one unapproved device. Send messages and make a host decision. Confirm only approved members receive the room event and history survives an app restart.

Icebreakers and continuity

Member experience: approved members see the interests they have in common with other approved members plus two conversation prompts. After attendance, they can opt into similar-event or circle suggestions.

How it works: the API calculates mutual interests, never exposes the other member's complete interest list, and respects the privacy toggle. Continuity choices feed persisted recommendations, not direct messages or a public follow graph.

Test: make two approved members share several interests, switch one member's icebreaker privacy off, refresh on the other account, and confirm that member disappears from icebreaker results.

Host creation, attendance, and feedback

Member experience: approved hosts can create/edit/cancel events and circles, decide member requests, record attendance/no-shows, and read aggregate feedback insights. Members can submit one post-event rating/note.

How it works: host permissions are checked on the API. Cancellation needs a reason and creates member notifications. Feedback is available after attendance and creates a one-time trust event; later edits do not award extra points.

Test: try creating as a normal member, then as an approved host. Record attendance, submit feedback twice, inspect trust events, and cancel an activity to verify notifications and audit records.

Notifications, settings, blocks, and privacy

Member experience: see a persisted notification inbox, control notification/recommendation/icebreaker settings, block hosts or members, and unblock them later.

How it works: notifications, settings, and blocks are PostgreSQL records. Blocks remove relevant discovery/recommendation interactions and prevent disallowed direct interactions. Native push delivery still needs the device-token integration described later in this guide.

Test: block a host then re-run discovery, change settings and restart, trigger an event notification, mark it read, and confirm the state is consistent across two sessions.

Admin dashboard and audit history

Operator experience: review selfie/host queues, look up members, search activities by city, correct locations, cancel activities, dispatch notifications, and inspect audit records.

How it works: the backend checks the temporary admin key today or a named Firebase admin role where configured. Every sensitive action is server-authorized and audited. Selfie viewer URLs are backend-generated and short-lived.

Test: use an unauthorized browser request, a low-privilege admin, and a correctly scoped admin. Confirm access fails/works exactly as intended and all changes have actor/time/action audit entries.

Report model: deliberately held

Member experience: there is currently no live report entry point in the member product.

How it works: reasons and API persistence exist for spam, fake profile, harassment, inappropriate behaviour, and other. The member entry points and admin moderation queue stay disabled until an operator workflow is staffed.

Test before enabling: submit each reason, verify reporter anonymity/authorization, assign and resolve in an admin queue, test suspension/appeal escalation, and measure response time.

4. Production Integrations, Exactly What Each One Does

3.1 Firebase Authentication

Why it exists

Firebase proves control of a phone number. It returns an ID token. Niva's backend verifies that token using Firebase Admin before it reads or changes private Niva data.

What to configure

Create Android and web Firebase app records; enable Phone sign-in; add the Android SHA-1/SHA-256 signing fingerprints; put Firebase client values in the mobile build environment; put the service-account values only in the API secret store.

What changes at release

Set EXPO_PUBLIC_AUTH_MODE=firebase and NIVA_BETA_AUTH_ENABLED=false. The local beta path then cannot create sessions in production.

  1. Member submits a phone number.Firebase performs its abuse protection and sends an SMS code. On supported Android devices, the optional PNV path can show a consent prompt first; SMS remains the universal fallback.
  2. Firebase verifies the code or native PNV result.The mobile app receives a Firebase ID token and refresh token. The refresh token is stored in the device's protected SecureStore, not in Niva's database.
  3. Niva exchanges the ID token for an API session.POST /auth/session verifies the Firebase signature server-side and creates/updates the PostgreSQL user/trust records. Every protected mobile API request includes a Firebase ID token.

Relevant code: apps/mobile/src/services/mobile-auth.ts, apps/mobile/src/services/firebase.ts, apps/backend/src/auth, and apps/backend/src/firebase.

Important distinction: standard Firebase SMS OTP is the required baseline. Android Firebase PNV, the native consent prompt that can avoid typing an OTP, is optional, Android-only, carrier-dependent, and should not be enabled until its Firebase billing, OAuth-brand verification, native build, and carrier tests are complete.

3.2 Firebase Storage For Profile Photos And Verification Selfies

Firebase Storage holds binary image files; PostgreSQL holds only Niva records and internal Storage paths. The client uploads to an owner-scoped path, then sends the internal path to the API. It does not send a public selfie URL.

3.3 PostgreSQL And Backend Hosting

Niva's API is the policy enforcement point. Firebase authenticates the member; NestJS decides whether she can join an event, access a chat, submit feedback, view icebreakers, or perform host/admin actions. PostgreSQL is the system of record for product data.

  • Provision managed PostgreSQL with encryption, automated backups, point-in-time recovery, separate production credentials, and a tested restore procedure.
  • Deploy the API on an HTTPS domain with a WebSocket-capable reverse proxy/load balancer. Socket.IO traffic must be supported, not only ordinary HTTP requests.
  • Run prisma migrate deploy against production as a controlled release step. Never use prisma migrate reset in production.
  • Set production CORS origins for the admin domain and any hosted web preview. Do not leave localhost origins enabled unnecessarily.
  • Store DATABASE_URL, Firebase Admin credentials, and the temporary admin bootstrap key in the host's secret manager, never in Git or EXPO_PUBLIC_* variables.
  • Add uptime checks, structured error reporting, database backup alerts, and access-log retention before inviting external users.

3.4 Expo Builds And Android Push Notifications

The repository now includes the complete notification code path: in-app records, realtime events, native permission and Expo-token registration, device-token records, delivery rows, an optional dispatch worker, Expo ticket validation, and stale-device deactivation.

Remaining integration item: create the EAS project ID, configure FCM V1 credentials, produce a development/release build, and validate permission denial, foreground/background delivery, and provider tickets on physical Android devices. Notification-tap deep links should be added when final navigation URLs are fixed.

  1. Set up an Expo account and run EAS configuration. Build profiles already exist in apps/mobile/eas.json.
  2. Download google-services.json for com.niva.niva, add an expo.android.googleServicesFile reference, and use the same Firebase project as Auth/Storage.
  3. Give EAS an FCM V1 service-account credential for push delivery. Keep the credential outside Git.
  4. Create an internal Android build, accept notification permission, register the token, trigger a notification, and confirm the provider receipt and the delivered notification.

3.5 Maps And Venue Search

Events persist a human-readable venue plus optional latitude/longitude. Hosts can explicitly request foreground location, the device reverse-geocodes the point, and activity details open exact directions. No continuous/background member location is collected.

  • Choose one provider and an expected monthly spend limit.
  • Create separate restricted keys for Android production, Android debug, backend services, and admin web if needed.
  • The MVP selector and migration-safe backend fields are implemented. Add provider-backed autocomplete/pin search only if host testing shows it is necessary.
  • Do not collect continuous member location for MVP. A user-selected event location is enough.

3.6 Admin Identity, Support, Monitoring, And Safety Operations

Named admins

The dashboard and API use Firebase-token administrators with persisted role checks and audit actors. Provision individual accounts and restrict NIVA_ADMIN_KEY to bootstrap/recovery because a shared key cannot identify a specific operator.

Monitoring

Integrate a crash/error service for the mobile app, API, and dashboard, plus uptime checks and alert routing. Keep personal data and selfie content out of error events and logs.

Support

Publish a support email and a simple support/deletion web page. Establish response expectations for verification status, event cancellation, safety incidents, and deletion requests.

Moderation

Before enabling reports, assign people who can triage reports, define severity/response times, document evidence handling, define temporary suspension rules, and test the moderator queue.

Secret rule: Firebase web/app configuration values may be exposed in an Expo client build because they identify the Firebase project, but they are not sufficient authorization. Because a Firebase Admin private key was shared in an earlier conversation, confirm that it has been revoked/rotated in Google Cloud IAM, remove any old local copies, and audit Git history and deployment secret stores before launch.

6. Test Properly Before Inviting Anyone

Local browser testing proves layout and basic API behavior. It does not prove the mobile services that users depend on. Use an internal Android build first, then a Play closed test.

5.1 Automated Release Gate

  • Backend build passes.
  • Backend Jest suite passes.
  • Mobile TypeScript check passes.
  • Admin production build passes.
  • Prisma migration deploy is tested on a staging database.
  • Database backup and restore has been rehearsed.
  • Dependency/security update review is completed.
  • Secrets scan finds no credentials in Git or mobile build variables.

5.3 Closed-Beta Operational Test

  1. Seed only enough real Bangalore activities for the first group. Avoid an empty app.
  2. Invite 10 to 20 trusted testers internally before a broader closed test. Observe authentication delivery, upload failures, join conversion, event attendance, chat behavior, and moderation load.
  3. Run at least one complete safety drill: selfie submitted, admin reviews it, user joins, host approves, event is cancelled, notification delivers, member requests help/deletion.
  4. Review error/crash reports daily and identify whether failures are from mobile, Firebase, network, API, or operator process.
  5. Only expand the cohort after the workflow, support responses, and data deletion handling are stable.

7. Google Play Store Release Requirements

Use Google Play testing tracks as part of the release process. A new app is uploaded as an Android App Bundle (AAB), not just an APK. EAS Build can produce the release artifact and manage signing credentials if chosen.

6.1 Prepare The Android Build

  1. Keep the already configured package name com.niva.niva unless a permanent rename is decided now. Package names are costly to change after publication.
  2. Set a production version and incrementing Android versionCode. Configure EAS build profiles for internal, preview/closed-test, and production release builds.
  3. Use a current Expo/React Native Android toolchain and verify the target SDK requirement shown in Play Console at submission time. Google Play blocks submissions that do not meet its current target API policy.
  4. Configure release signing through EAS or a controlled upload-key process. Store upload-key recovery/access details securely and restrict account access.
  5. Build the AAB, install it through the internal test track, and repeat the physical-device checks above before moving it to closed testing.

6.3 Closed Test Before Production

Use an internal track for the team, then a closed track for the actual beta cohort. For new personal developer accounts created after November 13, 2023, Google currently requires a closed test with at least 12 opted-in testers for 14 continuous days before production access can be requested. Check the account's Play Console dashboard because account type and policies can change.

Recommended rollout: internal build for team and operators, closed test for 20 to 50 consenting Bangalore members, then production only after safety operations, deletion, report handling, and reliability metrics are proven. A closed test is the right first release for Niva; it is not a lesser version of the product.

8. Recommended Order, One Manageable Stage At A Time

  1. Secure the project and choose production ownership.Rotate/audit credentials, create an Expo account and Play Console organization/account, establish a support email, and decide who owns Firebase, the database, and Play Console. Do this before copying production values anywhere.
  2. Make Firebase real.Configure Firebase Auth, Android app records/signing fingerprints, Storage bucket and rules, backend Admin secrets, and production mobile variables. Disable local beta auth. Validate authentication and private image flow on one physical Android build.
  3. Deploy a staging environment.Deploy API, PostgreSQL, dashboard, and policy/support site to HTTPS. Apply migrations, enable backups/monitoring, configure WebSocket support and CORS, and give admins named accounts.
  4. Validate the completed release code.Apply the new migration, provision named admins, configure EAS/FCM, and test push registration, deletion, foreground location, policy pages, and admin queues against staging. Keep report UI held only while the beta is small enough for manual safety handling.
  5. Run internal device tests.Use a release-like Android build. Test real SMS, selfie upload/review, session restoration, chat, host flows, notifications, privacy denial paths, and admin permissions. Fix issues before inviting external people.
  6. Publish the legal and operating foundation.Publish privacy policy, terms, community guidelines, selfie consent/retention rules, support/deletion page, moderation playbook, escalation process, and data-safety inventory.
  7. Start a Play closed test.Upload an AAB to internal testing, then create the closed test. Invite the first cohort, monitor daily, run an incident drill, collect feedback, and meet any account-specific test duration requirement.
  8. Request production access only after proof.Complete the Play Console declarations, use accurate screenshots and reviewer instructions, confirm deletion/report operations, and roll out gradually. Keep the first public availability geographically and operationally constrained.

9. Definition Of Done For The First Real Release

  • Production Firebase Phone Auth works on physical Android devices.
  • Beta authentication mode is disabled outside local development.
  • Firebase Storage rules are deployed and independently tested.
  • Verification selfies are reviewed only through authorized, expiring links.
  • Production PostgreSQL has tested backups and migrations.
  • API, admin, and support/policy pages run on HTTPS domains.
  • Named admin login replaces the shared admin key.
  • Native push registration and Android FCM delivery work.
  • Account deletion is available in app and on the web.
  • Privacy policy, Data safety declaration, consent, and retention rules match reality.
  • Report and moderation workflow is enabled before broad public availability.
  • Internal and closed tests pass on real devices and real networks.
  • Play Store AAB, store listing, screenshots, reviewer instructions, ratings, and audience declarations are complete.
  • Operators can respond to verification, cancellation, support, safety, and deletion requests.
