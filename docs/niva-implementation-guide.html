Niva closed-beta implementation

What is real today, how it works, and what is still needed.

Niva is now a database-backed closed-beta community app, not just a mocked mobile prototype. Core product decisions are enforced by the backend and persisted in PostgreSQL.

Local state: the local niva database is synchronized with the checked-in Prisma schema, including community continuity and beta analytics. The mobile app, API, and admin dashboard share the same records.

The product boundary

What Niva is

A controlled way for verified women in Bangalore to meet through small events and recurring circles.

What Niva is not

It is not a public social feed, a random-DM app, or a marketplace for anonymous hosts.

Why this matters

The safety model stays manageable in beta: verified joining, controlled chats, host review, and operator oversight. Reporting remains held until moderation operations are staffed.

End-to-end member flow

  1. Authenticate and complete a profileFirebase proves the phone identity in production. The backend upserts the local User and TrustProfile. Username, profile, self-declaration, and selfie review remain API-backed.
  2. Browse before verificationAnyone signed in can view published events and circles. There is no fake success state for placeholder activity cards.
  3. Verify before joiningA first join attempt from an unverified member opens the selfie review path. The server rejects join requests until verification is approved.
  4. Accept community guidelinesA verified member sees the women-centered community agreement before the first join. POST /users/me/community-guidelines records its version and timestamp. The join endpoint checks this again.
  5. Submit a join requestThe backend checks activity state, capacity, blocks, verification, and guidelines, then stores REQUESTED in EventMember or CircleMember.
  6. Host manages the activityFor events and circles, the actual host can approve or decline requests, edit live details, or cancel with a required reason. Saved changes and cancellations create persisted member notifications.
  7. Use cohort chatThe app shows approved event/circle chats only. Messages are saved in PostgreSQL and retrieved through guarded API routes. There are no direct messages.
  8. Close the loop after the eventThe host marks an approved member attended or no-show. The backend updates membership, writes a trust event, recalculates trust, and notifies the member. A participant can then leave a private one-to-five rating and optional note from My plans.

Host and admin responsibilities

Member

Completes profile and verification, agrees to guidelines, requests to join, chats only after approval, can block, leave feedback, set continuity choices, and view all mutual interests with approved cohort members.

Host

First needs trust eligibility plus an admin-approved HostApproval. Can create, edit, and cancel events/circles, approve/decline member requests, record attendance, and see anonymous feedback insights for her event.

Admin

Reviews selfies and host requests, and can cancel a published activity with a member-visible reason. Decisions create audit records. Current dashboard uses the beta key; backend also supports named Firebase-token admins.

Backend

Enforces permissions and owns all state: memberships, messages, held reporting records, approvals, feedback, notifications, trust events, and audit logs.

Notifications: what is real versus not yet live

In-app notifications are real today. Every relevant action creates a persistent Notification row and it appears in the mobile Notifications screen.

Push delivery has a real backend queue and Expo dispatch implementation. It is not yet sending to a physical phone because the mobile app does not register a native Expo push token.

  • Implemented notification preference, Notification rows, DevicePushToken model, NotificationDelivery queue, Expo dispatch endpoint.
  • Not live on devices expo-notifications, FCM/APNs credentials, and physical-device token registration. The backend queue worker can be enabled by deployment configuration.

Remaining work, in the right order

  1. Finish provider/device integrationsTest Firebase Phone Auth and Storage on physical devices; add Expo push token registration and FCM/APNs credentials.
  2. Harden admin accessAdd Firebase sign-in to the web dashboard, authorize actions by AdminRole, then retire the shared beta key.
  3. Refine host operationsAdd a validated map picker, platform-native date/time picker, richer activity search, and decide whether feedback should ever affect trust. Current date/time controls are intentionally constrained day and 30-minute adjustments.
  4. Prepare public-launch operationsWrite privacy, selfie retention/deletion, moderation escalation, support, incident-response procedures, and the workflow required before report UI is re-enabled.
